# Инструкция для методологов по работе со справочником навыков

## Для кого этот документ

Этот документ написан для методологов и экспертов, которые работают со справочником навыков и индикаторов, но не занимаются программированием и не обязаны понимать устройство базы данных.

Ниже описано:

- что такое группа, скилл и индикатор;
- как проверять импортированные данные;
- как добавлять и исправлять содержимое справочника;
- как не допускать дублей и путаницы;
- в каком порядке лучше работать.

Документ относится к локальному рабочему контуру справочника, который используется для наполнения, проверки и подготовки данных перед дальнейшей публикацией.

## Что такое справочник простыми словами

Справочник состоит из трех уровней:

1. `Группа skills`
2. `Skill`
3. `Indicator`

Простой пример:

- Группа: `Работа с данными`
- Skill: `Написание SQL-запросов`
- Indicator: `Знает основы SELECT, JOIN, GROUP BY`
- Indicator: `Умеет писать запросы с фильтрацией и агрегацией`

То есть:

- группа объединяет близкие навыки по теме;
- скилл описывает отдельный навык;
- индикатор показывает, что именно человек должен знать или уметь по этому навыку.

## Что означают основные термины

### Группа

Группа нужна для удобной навигации. Это крупный тематический раздел.

Примеры:

- `Алгоритмы`
- `Работа с инфраструктурой`
- `Разработка веб-приложений`

### Skill

Скилл это отдельный навык внутри группы. Он должен быть достаточно конкретным, чтобы по нему можно было написать понятные индикаторы.

Хороший пример названия skill:

- `Работа с хеш-таблицами`
- `Использование Docker`
- `Настройка CI/CD`

Нежелательный пример:

- `Разработка`
- `Работа с кодом`
- `Современные технологии`

Слишком широкие формулировки затрудняют проверку и наполнение.

### Indicator

Индикатор это конкретное проявление навыка. По индикатору должно быть понятно, что именно требуется от человека.

Хороший индикатор:

- `Знает назначение индексов в базе данных`
- `Умеет писать Dockerfile для приложения`

Слабый индикатор:

- `Понимает тему`
- `Работает хорошо`
- `Может использовать`

Если индикатор слишком общий, его трудно проверять и применять в профилях и проектах.

Что где делать:

- в `Архиве` удобно восстанавливать ранее скрытые записи;
- в `Проверке` удобно разбирать спорные или неполные записи после импорта.

## Что такое архив

Архив это отдельный экран, куда попадают записи, которые были убраны из рабочего каталога без физического удаления.

Архив нужен, чтобы:

- не терять данные;
- иметь возможность вернуть запись обратно;
- безопасно исправлять ошибки наполнения.

В архив могут попадать:

- группы;
- skills;
- индикаторы.

### Когда запись попадает в архив

Обычно архив используется в тех случаях, когда запись уже содержит вложенные данные и удалять ее полностью рискованно.

Примеры:

- группа уже содержит skills;
- skill уже содержит индикаторы;
- индикатор пришел из импорта и связан с исходным материалом.

### Что можно делать в архиве

В архиве можно:

- посмотреть, какие записи были скрыты;
- восстановить группу;
- восстановить skill;
- восстановить индикатор.

После восстановления запись снова появляется в рабочем каталоге.

## С чего начинать работу

Рекомендуемый порядок такой:

1. Открыть страницу `Проверка`
2. Разобрать спорные записи
3. Проверить, все ли skills попали в правильные группы
4. Проверить формулировки названий skills
5. Проверить индикаторы внутри каждого skill
6. Только после этого добавлять новые записи

Это помогает сначала привести в порядок уже существующий каталог, а потом расширять его.

## Как работать с очередью проверки

Страница `Проверка` нужна для ручной валидации спорных данных, которые пришли из Excel или других источников.

На этой странице можно:

- посмотреть проблемную запись;
- понять, в чем именно сомнение;
- оставить комментарий;
- подтвердить запись;
- пропустить запись;
- вернуть запись в работу.

### Что означают статусы

- `open` — запись еще не разобрана;
- `resolved` — запись проверена и решение принято;
- `ignored` — запись решено не использовать.

### Когда ставить `resolved`

Статус `resolved` ставим, если:

- вы поняли, к какому skill относится запись;
- исправили или подтвердили формулировку;
- запись больше не требует обсуждения.

### Когда ставить `ignored`

Статус `ignored` ставим, если:

- это дубликат;
- это служебная строка из шаблона;
- это случайный мусор из исходного файла;
- запись не нужна для каталога.

### Когда возвращать в `open`

Возвращаем в `open`, если:

- запись открыли слишком рано;
- по ней нужно дополнительное согласование;
- появились новые обстоятельства и нужно пересмотреть решение.

### Как писать комментарий

Комментарий должен быть коротким и понятным.

Хорошие примеры:

- `Дубль skill "Работа с Docker", отдельную запись не создаем`
- `Исправили формулировку, оставили в группе "Инфраструктура"`
- `Это шаблонная строка, в каталог не берем`

Нежелательные комментарии:

- `ok`
- `норм`
- `посмотрели`

Комментарий нужен, чтобы другой человек потом понял, почему было принято такое решение.

## Как создать новую группу

Новую группу имеет смысл создавать только тогда, когда новый skill явно не укладывается ни в одну существующую группу.

### Когда группу создавать стоит

- появилась новая большая предметная область;
- внутри нее ожидается несколько skills;
- эта область будет понятна пользователям справочника.

### Когда группу создавать не стоит

- если речь идет всего об одном skill;
- если skill можно отнести к уже существующей группе;
- если новая группа отличается от существующей только формулировкой.

### Правила для названия группы

Название группы должно быть:

- коротким;
- понятным;
- без лишних деталей;
- без аббревиатур, если без них можно обойтись.

Хорошие примеры:

- `Работа с данными`
- `Архитектура`
- `Командная работа`

Нежелательные примеры:

- `Разное`
- `Важные вещи`
- `Инфраструктурные и операционные аспекты жизненного цикла`

### Если группа создана по ошибке

В рабочем контуре действует безопасное правило:

- если группа пустая, ее можно удалить полностью;
- если внутри группы уже есть skills, такая группа не удаляется физически, а переводится в архивный статус.

Почему это сделано так:

- пустую группу можно убрать без риска;
- непустую группу опасно удалять сразу, потому что вместе с ней можно потерять skills и индикаторы.

Поэтому если группа была создана по ошибке:

1. Если внутри еще ничего нет, удаляйте ее
2. Если внутри уже есть skills, используйте архивирование
3. Если нужно потом переразложить skills по другим группам, это делается отдельно и аккуратно

## Как создать новый skill

Skill создается внутри выбранной группы.

Перед созданием нового skill обязательно проверьте:

- нет ли уже такого skill в этой группе;
- нет ли похожего skill в другой группе;
- не отличается ли новый skill только формулировкой или опечаткой.

### Хороший порядок действий

1. Открыть нужную группу
2. Просмотреть уже существующие skills
3. Убедиться, что дубля нет
4. Создать новый skill
5. Сразу добавить к нему индикаторы

### Как назвать skill

Название должно быть:

- конкретным;
- коротким;
- одинаково понятным для всех участников;
- без лишней описательности.

Хорошие примеры:

- `Работа с REST API`
- `Использование Docker`
- `Работа с потоками в Java`

Нежелательные примеры:

- `Знание современных подходов к работе с REST API`
- `Умение использовать контейнеризацию в практической деятельности`
- `Владение разными приемами многопоточной разработки`

Такие формулировки лучше разбивать:

- короткое название skill;
- подробности переносить в индикаторы.

### Что такое порядок skill

У каждого skill есть поле порядка. Оно отвечает только за место skill внутри группы.

Практически это значит:

- skill с порядком `1` будет выше;
- skill с порядком `2` будет ниже;
- и так далее.

Порядок полезен, если вы хотите выстроить skills:

- от базовых к более сложным;
- от общих к специальным;
- в привычной для методологов логике.

## Как понимать сложность

В справочнике теперь используется и признак сложности.

Сложность помогает понять:

- на каком уровне обычно появляется skill;
- насколько сложен конкретный индикатор;
- как skill развивается от более простого уровня к более сложному.

### Где сложность видна

Сложность показывается:

- у skill как общий диапазон;
- у индикатора как конкретный уровень.

Примеры:

- `Начальный (junior-)`
- `Базовый (junior+)`
- `Продвинутый (middle)`
- `Мастерский`

### Как трактовать сложность у skill

Если у skill указана одна сложность, это значит, что его индикаторы в основном относятся к одному уровню.

Если у skill указан диапазон, например:

- `Начальный (junior-) -> Базовый (junior+)`

это значит, что внутри skill есть более простые и более сложные индикаторы.

### Как трактовать сложность у индикатора

Сложность индикатора показывает, на каком уровне обычно ожидается именно это знание или умение.

Примеры:

- `Знает основы Docker` — чаще базовый или начальный уровень
- `Умеет оптимизировать JVM` — чаще более продвинутый уровень

### Как использовать сложность в работе

Сложность нужна не для красоты, а для практической пользы:

- при наполнении профилей;
- при обсуждении требований к ролям;
- при проверке баланса skills внутри группы;
- при построении логики от простого к сложному.

### Важное правило

Сложность skill считается по его индикаторам.

Это значит:

- если вы меняете сложность индикаторов, общий уровень skill тоже может измениться;
- если skill новый и индикаторов у него еще нет, сложность может пока не отображаться.

## Как редактировать существующий skill

Редактировать стоит в трех случаях:

- нужно исправить неточность;
- нужно убрать дублирование;
- нужно сделать название понятнее.

### Когда лучше редактировать, а не создавать новый

Нужно редактировать существующий skill, если:

- смысл тот же, а отличается только формулировка;
- в названии есть опечатка;
- skill уже используется в каталоге, но название можно улучшить.

Новый skill лучше не создавать, если старый можно поправить без потери смысла.

### Когда нужно остановиться и согласовать

Нужно отдельно согласовать правку, если:

- меняется смысл skill;
- один skill хотят разделить на два;
- два skills хотят объединить в один;
- меняется логика группировки.

Это уже не простая редактура, а содержательное изменение модели справочника.

### Если skill создан по ошибке

Для skill действует безопасное правило:

- если в skill еще нет индикаторов, его можно удалить полностью;
- если в skill уже есть индикаторы, такой skill не удаляется физически, а архивируется.

Это сделано для того, чтобы не потерять уже собранное содержание.

Практически это значит:

1. Пустой skill можно удалить
2. Наполненный skill лучше архивировать
3. Если потом нужно перенести индикаторы в другой skill, это делается отдельно

## Как писать хорошие индикаторы

Индикаторы это самая важная часть справочника. Именно они показывают, что стоит за skill на практике.

### Хороший индикатор должен быть

- понятным;
- наблюдаемым;
- проверяемым;
- без лишней воды.

### Хорошие примеры

- `Знает назначение Dockerfile и docker-compose`
- `Умеет собирать контейнер приложения`
- `Знает принципы работы транзакций`
- `Умеет писать SQL-запросы с JOIN`

### Слабые примеры

- `Разбирается в теме`
- `Работает с инструментом`
- `Понимает, как использовать`

Такие индикаторы лучше переписать так, чтобы было ясно, что именно должен знать или уметь человек.

### Какой тип индикатора выбирать

В текущем рабочем контуре используются типы:

- `Знает`
- `Умеет`
- `Владеет`
- `Не указано`

Рекомендуемая логика:

- `Знает` — когда речь о понятиях, принципах, устройстве, назначении;
- `Умеет` — когда речь о действии, применении, настройке, разработке;
- `Владеет` — когда речь о устойчивом практическом использовании и уверенном применении;
- `Не указано` — только если тип пока неясен и требует уточнения.

По возможности лучше не оставлять индикатор в статусе `Не указано`, а уточнить его тип.

### Если индикатор создан по ошибке

Для индикаторов правило зависит от происхождения записи.

- ручной индикатор можно удалить полностью;
- импортированный индикатор не удаляется физически, а архивируется.

Почему так:

- ручной индикатор легко убрать без риска;
- импортированный индикатор связан с исходным материалом, поэтому его безопаснее выключить, а не уничтожить.

Если вы сомневаетесь, какой это индикатор, безопаснее сначала архивировать, а потом уже принимать решение после обсуждения.

### В каком порядке располагать индикаторы

Лучше придерживаться простой логики:

1. Сначала `Знает`
2. Потом `Умеет`
3. Потом `Владеет`

Внутри каждого типа можно располагать формулировки:

- от более простых к более сложным;
- от базовых к продвинутым;
- от общего к частному.

## Правила качества справочника

Перед тем как считать раздел приведенным в порядок, проверьте следующие вопросы.

### Для группы

- название понятно без пояснений;
- группа не дублирует соседнюю;
- skills внутри группы действительно относятся к одной теме.

### Для skill

- skill не дублирует существующий;
- название короткое и предметное;
- skill действительно является отдельным навыком, а не целой темой.

### Для индикаторов

- индикаторы не повторяют название skill слово в слово;
- индикаторы не слишком общие;
- индикаторы не противоречат друг другу;
- из индикаторов понятно, как проявляется skill.

## Частые ошибки

### Ошибка 1. Создание дубля вместо правки

Пример:

- уже есть skill `Работа с Docker`
- создается новый skill `Использование Docker`

В такой ситуации чаще всего нужен не новый skill, а проверка и унификация названия.

### Ошибка 2. Слишком широкое название

Пример:

- `Работа с инфраструктурой`

Это больше похоже на группу, а не на отдельный skill.

### Ошибка 3. Индикатор слишком общий

Пример:

- `Умеет работать с системой`

Лучше заменить на конкретное действие.

### Ошибка 4. Смешение разных навыков в одном skill

Пример:

- `Docker, Kubernetes и CI/CD`

Если внутри одного названия скрыто несколько отдельных тем, лучше разделить их на разные skills.

### Ошибка 5. Использование служебных пометок в финальном названии

Нежелательно оставлять в названии:

- черновые пометки;
- сокращения, понятные только одному человеку;
- служебные комментарии;
- фразы вроде `проверить`, `временное`, `из Excel`.

## Рекомендуемый рабочий сценарий

Ниже самый практичный сценарий повседневной работы.

### Сценарий 1. Разобрать спорные записи после импорта

1. Открыть `Проверка`
2. Выбрать записи со статусом `open`
3. Посмотреть причину
4. Найти подходящую группу и skill
5. Принять решение
6. Оставить комментарий
7. Поставить `resolved` или `ignored`

### Сценарий 2. Добавить новый skill в существующую группу

1. Открыть нужную группу
2. Проверить, нет ли дубля
3. Создать skill
4. Указать порядок
5. Добавить индикаторы
6. Проверить формулировки

### Сценарий 3. Исправить существующий skill

1. Найти skill
2. Понять, это правка формулировки или изменение смысла
3. Если меняется только формулировка, сохранить правку
4. Если меняется смысл, сначала согласовать

### Сценарий 4. Добавить индикатор

1. Открыть нужный skill
2. Проверить, нет ли уже похожего индикатора
3. Выбрать тип
4. Написать конкретную формулировку
5. Указать порядок
6. Сохранить

## Что безопасно делать без технической команды

Можно самостоятельно:

- редактировать формулировки;
- упорядочивать skills внутри группы;
- добавлять недостающие индикаторы;
- закрывать явные дубли и шаблонные записи в `Проверке`;
- исправлять опечатки;
- улучшать ясность названий.

## Что лучше делать после согласования

Лучше отдельно согласовывать:

- создание новой большой группы;
- объединение нескольких skills в один;
- разделение одного skill на несколько;
- массовое переименование;
- удаление большого количества индикаторов;
- изменения, которые влияют на логику всего справочника.

## Короткая памятка

Если нужно запомнить только главное, то правило такое:

1. Сначала ищем дубль
2. Потом редактируем существующее
3. Новый skill создаем только если старого действительно нет
4. Индикаторы пишем конкретно и понятно
5. Спорные случаи не угадываем, а фиксируем комментарием и согласовываем

## Итог

Цель работы со справочником не в том, чтобы просто перенести текст из Excel, а в том, чтобы получить чистый, понятный и единообразный каталог навыков.

Хороший справочник это такой справочник, в котором:

- легко найти нужный skill;
- нет дублей и мусора;
- индикаторы понятны без дополнительных объяснений;
- разные методологи принимают похожие решения по одним и тем же правилам.

Если потребуется, на основе этого документа можно сделать еще две отдельные версии:

- короткую памятку на 1 страницу;
- расширенный регламент с примерами по конкретным ролям и предметным областям.
