# Аудит учебного контента

Рабочий прототип инструмента, который принимает локальную папку учебного проекта,
находит проверяемые материалы и формирует отчёт по критериям из ТЗ.

## Быстрый запуск

```powershell
$env:PYTHONPATH = "src"
python -m content_audit --input .\proj_example --output .\reports\proj_example --skip-network
```

## Веб-интерфейс

```powershell
$env:PYTHONPATH = "src"
python -m content_audit.web_app --host 127.0.0.1 --port 8021
```

После запуска откройте [http://127.0.0.1:8021](http://127.0.0.1:8021).
На вход подаётся путь к папке проекта, на выходе показывается панель отчёта
и ссылки на файлы выгрузки.

Результаты:

- `report.json` — полный машинный отчёт;
- `report.csv` — таблица для методологов;
- `run_summary.json` — краткая сводка прогона.

## Что уже проверяется без модели

- доступность локальных файлов и структура единицы контента;
- ссылки в Markdown;
- наличие и базовая связность `check-list.yml`;
- наличие языковых версий материалов;
- наличие признаков финальной проверки;
- качество локальных изображений по размеру;
- незавершённые плейсхолдеры и базовая читаемость текста;
- наличие лицензии;
- кандидаты на проверку актуальности технологий, версий и дат.

## Модельные проверки

Если задан `OPENROUTER_API_KEY` или `OPEN_ROUTER_API_KEY`, можно включить
модельный контур. Модель можно задать через `OPENROUTER_MODEL`,
`OPEN_ROUTER_MODEL` или параметр `--openrouter-model`.

Дополнительно можно задать отдельные модели:

- `OPENROUTER_FACT_MODEL` или `OPEN_ROUTER_FACT_MODEL` — фактологическая проверка через поисковую модель Perplexity, по умолчанию `perplexity/sonar`;
- `OPENROUTER_TECH_MODEL` или `OPEN_ROUTER_TECH_MODEL` — проверка актуальности технологий, по умолчанию `qwen/qwen3-coder`.

```powershell
$env:PYTHONPATH = "src"
$env:OPENROUTER_API_KEY = "<key>"
python -m content_audit --input .\proj_example --output .\reports\proj_example --use-model --skip-network
```

Модельный контур отделён от обычной логики: извлечение, проверка, оценка
критичности и рекомендации остаются отдельными шагами.

Повторные проверки одинаковых фактов и технологий берутся из `audit_cache.json`
в папке отчёта. В CSV и JSON для внешних проверок есть отдельные поля:
источник, дата проверки, статус поддержки, последняя версия и рекомендуемая
версия.

Сводка также хранит длительность шагов, версии модельных запросов и статистику
модельных вызовов: число свежих запросов, попадания в кэш, токены и стоимость,
если OpenRouter вернул эти поля.

## Ссылки и безопасность

Сетевые проверки включаются только без `--skip-network`. Локальные адреса,
внутренние IP и ссылки с учётными данными не проверяются автоматически.

## Метрики качества

Когда появится размеченная выборка, можно записать `evaluation.json`:

```powershell
python -m content_audit --input .\proj_example --output .\reports\proj_example --gold .\gold.json
```

Метрики считают точность, полноту, полноту по критическим случаям и долю ложных
срабатываний по ключу `unit_id + criterion + file_path + line_start`.
