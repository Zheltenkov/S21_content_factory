## Введение

При создании приложения для миллионов пользователей и обрабатывающего десятки или даже тысячи запросов в секунду, нужно соблюдать ряд требований. Данные требования позволяют достичь следующих целей:

- не "упасть" при неожиданном повышении нагрузки - числа входящих запросов
- не "положить" смежные сервисы - поставщики данных, не создавать лишнюю нагрузку на систему в целом
- вернуть осмысленный ответ клиенту даже в случае сбоя
- обработать максимальное число пользовательских запросов на имеющемся "железе"

Назовем эти требования паттернами надежности. Именно их ты изучишь в этом проекте:

1. Оптимальное использование памяти
2. Пул потоков
3. Обработка ошибок
4. Timeouts
5. Rate Limiter
6. Circuit Breaker
7. Retry
8. Fallback
9. Cache

В папке Materials собраны материалы, на которые можно ориентироваться при работе над проектом.

## Общие замечания

**Тестирование паттернов**

Некоторые паттерны для проверки работы - вручную или через тест - требуют метода в классе REST контроллера. Рекомендуется для каждого паттерна делать отдельный метод. Так можно избежать их взаимного влияния и упростить тестирование. Но, т.к. в реальной жизни бизнес операция для выполнения требований надежности может потребовать настройки нескольких паттернов, такую реализацию предлагается сделать и протестировать на отдельном методе как "задачу со звездочкой".

**Версии библиотек**

В задании не указываются конкретные версии библиотек, т.к. со временем любая версия устареет. Это справедливо всегда, кроме тех случаев, когда самостоятельный подбор работающей версии занимает много времени.

Рекомендации актуальны на 2025 год.

### Задание 0. Тестовый бизнес-сервис

**_Представь, что ты разрабатываешь сервис по рекомендации игр. Задача - вернуть клиенту список рекомендованных игр по REST API._**

**_Для формирования списка сервис использует два внешних REST-вызова:_**

1. **_вызов сервиса рекомендаций, в ответ возвращается список id игр_**
2. **_вызов сервиса игр для получения данных об игре по id, полученным от сервиса рекомендаций_**

В IntelliJ Idea создай новый проект:

- Выбери Spring Boot проект.
- Выбери систему сборки Gradle.
- Выбери JDK 21.
- Выбери для Gradle DSL - Kotlin.

Важный момент - при промышленной разработке рекомендуется использовать LTS (Long Term Support) релизы Java. Т.е. релизы с долговременной поддержкой. Это означает, что исправления ошибок (patches) для данной версии будут выпускаться существенно дольше.

Подробнее <https://www.oracle.com/java/technologies/java-se-support-roadmap.html>

Основные шаги

1. создай структуру проекта с разделением по разным пакетам слоя представления - web и слоя адаптеров к внешним сервисам - adapter. Модель можно не выносить в отдельный корневой пакет, т.к. твой тестовый сервис не содержит много бизнес-логики, можно обойтись паттерном Transaction Script, и вызывать код адаптеров непосредственно из контроллера.

Возможную схему пакетов можно посмотреть в misc/arch.plantuml.

Слой mapper опционален и может быть полезен при преобразовании DTO, полученных из слоя адаптеров.

1. создай REST endpoint используй Spring Web REST Controller
2. напиши код адаптеров для вызова сервиса рекомендаций и сервиса игр с помощью RestClient или WebClient в блокирующем режиме
3. напиши код получения списка рекомендованных игр в контроллере - вызов адаптеров, маппинг полученных данных в модель, используемую в вашем API
4. создай заглушку для сервисов рекомендаций и игр, например, с помощью WireMock или mockwebserver. Рекомендуемые версии библиотек: "com.github.tomakehurst:wiremock-standalone:3.0.1" и "com.squareup.okhttp3:mockwebserver:4.12.0". Заглушка не должна запускаться при промышленной эксплуатации, поэтому реализуй ее запуск под DEV-MODE профайлом Spring. Заглушку можно использовать как при отладке, так и для интеграционных тестов.

Оффтоп: требования надежности особенно важны для сервисов в больших (enterprise) компаниях. Большая компания = много сервисов. В условиях господства микросервисной архитектуры у сервиса скорее всего будет несколько сервисов-контрагентов.

При этом нет возможности держать все эти сервисы в рабочем состоянии для отладки. А корректную работу вашего кода нужно подтвердить, причем в режиме, максимально приближенном к боевому. Поэтому приходится использовать заглушки.

1. напиши интеграционный тест, проверяющий успешный вызов нашего endpoint, используя Spring Test. Для тестирования вызова нашего сервиса можно использовать MockMvc, RestAssured или TestRestTemplate. Для приложений со сложной логикой основной упор нужно делать на написание unit тестов. Но паттерны надежности можно проверить с помощью интеграционных тестов.

### Задание 3. Обработка ошибок

При работе любого приложения возникают исключения - Exceptions. Ошибки могут быть как в коде, так и возникать при обращении к внешним сервисам.

Исключения выглядят примерно так:

```java
"status": 500, "error": "Internal Server Error", "trace": "org.springframework.web.reactive.function.client.WebClientResponseException: 200 OK from GET <http://localhost:8090/recommendations/list>, but response failed with cause: io.netty.handler.timeout.ReadTimeoutException at org.springframework.web.reactive.function.client.WebClientResponseException.create(WebClientResponseException.java:337) at org.springframework.web.reactive.function.client.DefaultClientResponse.lambda$createException$1(DefaultClientResponse.java:214) at reactor.core.publisher.FluxMap$MapSubscriber.onNext(FluxMap.java:106)
```

На первый взгляд, в данном ответе много полезной информации, даже есть stacktrace. Но она необходима разработчику. Потребитель сервиса ее видеть не должен.

Второй важный момент - API должно быть предсказуемым, т.е. при любых исключениях ответы с информацией об ошибке должны быть единообразны.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

1. _Почему потребитель нашего API не должен видеть имя исключения и stacktrace?_
2. _Как мы можем передать информацию об ошибке, возникшей при обработке запроса к нашему API, потребителю?_
3. _Какие средства для обработки ошибок предоставляет Spring Framework?_

**Результат**

1. Создай endpoint выбрасывающий исключение, посмотрите, какой ответ вернется клиенту
2. Исключи отображение в ответе информации, раскрывающей внутренности вашего сервиса
3. Продумай, как ты будешь возвращать ошибки потребителю.
4. Напиши тест, проверяющий успешный вызов и вызов с ошибкой.

### Задание 4. Timeouts

Обработка запроса клиента занимает некоторое время. Иногда это время может быть больше, чем мы ожидаем.

При этом, клиент, как правило, не готов ждать долго, чтобы получить запрошенную информацию.

Представь ситуацию: сервис-поставщик начинает долго отвечать, соответственно, обработка запроса твоим сервисом занимает больше времени. В **результат**е часть клиентов, не дождавшись ответа, закрывает сайт или приложение, в то время как от новых клиентов продолжают поступать запросы.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

1. _Чем отличается connection timeout от request timeout?_
2. _К чему может привести ситуация, когда таймаут очень большой или что еще хуже - не ограничен?_
3. _Как настроить таймауты для выбранного вами ранее http client?_
4. _Можно ли настроить таймаут для endpoint? Где это лучше сделать - в самом сервисе или на прокси-сервере перед ним?_
5. _Как должны соотносится таймаут для входящего запроса к твоему сервису и таймауты на запросы к сервисам, поставщикам данных?_
6. _Подумай, как определить оптимальную величину таймаута? Предупреждение: однозначного ответа на этот вопрос нет._

**Результат**

1. Настрой явно connection timeout и request timeout для выбранного http client
2. Вынеси настройки таймаута в файл с настройками Spring Boot, это упростит их изменение в будущем
3. Сэмулируй долгий ответ одного из твоих сервисов-поставщиков.
4. Пробрось информацию об ошибке таймаута с учетом знаний из предыдущего задания
5. Напиши интеграционный тест, проверяющий данный кейс

### Задание 5. Rate Limiter

Количество запросов к сервису может меняться со временем. Причиной может быть как поток новых клиентов после удачной рекламы, DDOS атака или даже ошибка в коде сайта или мобильного приложения.

Даже если запросы не вредоносные, не всегда есть возможность масштабировать серверные мощности и обработать все поступившие запросы.

В таком случае требуется настройка Rate Limiter.

Важно информировать клиента о том, что его запрос отбит Rate Limiter-ом.

Рациональное поведение потребителя в этом случае - подождать некоторое время перед повтором.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

1. _Для чего нужен Rate Limiter?_
2. _Почему лучше "отбить" часть запросов клиентов, чем попытаться обработать все?_
3. _Предположим, Rate Limiter настроен с периодом 1 секунда и rate равным 10. Рассмотрим интервал 2 секунды. Пусть за последние 100 мс первой секунды к нам пришло 10 запросов, и в первые 100 мс второй секунды - тоже 10 запросов. Как отработает Rate Limiter? В каком случае мы можем пропустить такой всплеск запросов, в каком - должны отбить?_
4. _Как настроить Rate Limiter, если у нас максимально допустимая частота запросов 0.2 запроса в секунду?_
5. _Какой код ошибок HTTP используется для возврата ошибки клиенту при срабатывании Rate Limiter?_
6. _Предположим, вы предоставляете внешнее API с помощью API шлюза, который расположен перед вашим приложением. Где лучше разместить Rate Limiter? Плюсы и минусы обоих вариантов?_
7. _В каких еще случаях, кроме защиты от перегрузки сервера, может быть полезен Rate Limiter ?_

**Результат**

1. Выбери и добавь в проект библиотеку, обеспечивающую функционал Rate Limiter. Например, resilience4j
2. Установи ограничение числа запросов на метод контроллера
3. Напиши интеграционный тест, приводящий к срабатыванию Rate Limiter.
4. Проверьте, что часть запросов обработалась успешно, а по остальным вернулась ошибка. Проверьте код ошибки
5. Вынеси настройки Rate Limiter в файл с настройками Spring Boot

### Задание 6. Circuit Breaker

Сервис, с которыми ты взаимодействуешь, может "упасть". Рассмотрим вариант, когда сервис-контрагент не отдает ошибку сразу, а просто ничего не возвращает.

После настройки Timeout, подожди N секунд, после чего запрос отвалится по таймауту. Вроде бы все хорошо. Но если ты знаешь по предыдущим неудачным запросам, что сервис не отвечает, можно не заставлять новых клиентов ждать.

Более того, даже если сервис отдает ошибку практически сразу, все равно мы тратим время на взаимодействие по сети, которое всегда на порядок дольше, чем вызов внутри программы.

Второй важный момент - пусть поставщик не упал, а еще жив, но "подтормаживает" - т.е. ему не хватает ресурсов, и он обрабатывает запросы сильно дольше, чем все готовы ждать.

В этом случае стоит дать ему восстановиться, а не "добивать" его запросами.

Все эти задачи решает Circuit Breaker.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

### Задание 7. Retry

Как уже говорилось, сервисы, с которыми мы взаимодействуем, могут быть недоступны.

Однако не всегда ошибка при запросе к сервису означает, что сервис не работает и к нему нет смысла обращаться в ближайшее время. Это может быть временная проблема в сети, например, связанная с обновлением прокси-сервера, или сработал Rate Limiter, и конкретно нашему запросу не повезло.

В таких случаях имеет смысл повторить запрос, но не сразу, а спустя некоторое время.

Время повторной попытки с одной стороны ограничивается максимальным временем, в течение которого **результат** еще будет актуален, а с другой - минимальным временем, необходимым для устранения временной проблемы, вызвавшей сбой.

Механизм повторных попыток (Retry) можно реализовать самостоятельно, однако существуют готовые решения и компоненты, которые значительно упрощают эту задачу.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

2. _Какие параметры мы можем настроить для алгоритма повторов?_
3. _Что такое exponential backoff? Сравните его с обычным повтором_
4. _Как выбрать минимальный интервал для начала повторов?_
5. _Почему важно ограничить число попыток?_
6. _При всех ли ошибках стоит использовать Retry? 403 forbidden? 429 Too Many Requests ?_

**Результат**

1. Выбери и добавь в проект библиотеку, реализующую механизм повторов. Подсказка - Spring Framework предоставляет такой функционал
2. Настрой повторы при возникновении ошибок при запросе к сервису рекомендаций
3. Настрой список ошибок, при которых будет срабатывать Retry
4. Настрой заглушку так, чтобы сервис рекомендаций возвращал 2 ошибки, а потом - успешный ответ.
5. Напиши интеграционный тест, приводящий к запуску повторов
6. Убедись, что с третьей попытки получен успешный ответ
7. Вынеси настройки Retry в файл с настройками Spring Boot

### Задание 8. Fallback

Снова поговорим о сбоях.

Предположим, твой сервис не смог обработать запрос клиента. Это может быть вызвано как ошибкой в самом сервисе, так и сбоем у смежного сервиса.

Что делать в таком случае? Самый простой вариант - вернуть ошибку. Но всегда ли это единственный возможный ответ? Бывают ситуации, когда можно вернуть клиенту частичный **результат**, взять ответ из кэша или хотя бы предоставить рекомендации по дальнейшим действиям.

Именно это и реализует паттерн Fallback - запасной механизм, который позволяет системе возвращать альтернативный или частичный ответ, если основной сервис недоступен или произошел сбой. Паттерн повышает надежность и удобство для пользователя, предотвращая резкие провалы, и помогает сохранить работоспособность сервиса даже при временных ошибках.

**ВОПРОСЫ, КОТОРЫЕ ТОЧНО ЗАДАСТ ТВОЙ ТИМЛИД.**

1. _Для чего нужен Fallback?_
2. _Что можно вернуть клиенту при отсутствии актуальных данных?_
3. _В каких случаях нельзя вернуть клиенту альтернативный содержательный ответ вместо ошибки?_

**Результат**

1. Изучи, как реализован fallback механизм в Spring Retry и resilience4j
2. Реализуй fallback ответ в виде возврата рекомендации по умолчанию при падении сервиса рекомендаций
3. Настройки для рекомендации по умолчанию вынеси в файл с настройками Spring Boot
4. Добавь возможность отключения fallback ответа, вынеси рубильник в файл настроек Spring Boot
5. Напиши интеграционный тест на проверку fallback ответа

### Задание 9. Cache

Часто запросы клиентов повторяются. Клиент может неосознанно или сознательно нажать кнопку «Обновить», а ответы для разных клиентов могут быть одинаковыми.

В таких случаях всегда ли имеет смысл обращаться к сервису-контрагенту за данными? Ведь эти данные обычно хранятся в базе данных, а запросы к БД - это “тяжёлые” операции, которых стоит избегать без необходимости. Кроме того, как уже говорилось, сетевой запрос всегда добавляет лишние задержки.

С другой стороны, полученный ответ может быть актуален через секунду или даже несколько минут, но не гарантировано, что он останется таковым через час.

Избежать избыточных запросов помогает механизм кэширования данных (cache), который позволяет повторно использовать недавно полученную информацию и значительно снижать нагрузку на сервисы и базы данных.

3. _Какие данные стоит кэшировать? Рекомендации? Курс валют? Цену на товар?_
4. _Как связано кэширование с реализацией паттерна Fallback?_
5. _В чем разница между локальным и распределенным кэшом? В чем их плюсы и минусы?_
6. _От чего зависит величина TTL (Time To Live) для записи в кэше?_

**Результат**

1. Выбери и подключи библиотеку, реализующую локальное кэширование (для простоты в этом проекте мы реализуем локальный кэш). Важным преимуществом библиотеки в нашем случае является простота ее интеграции с механизмом кэширования Spring Boot, как, например, у Caffeine. Рекомендуемая версия библиотеки: "com.github.ben-manes.caffeine:caffeine:3.1.8"
2. Реализуй кэширование ответов сервиса с информацией об играх
3. Настрой заглушку, чтобы она вернула один успешный ответ с информацией по игре, а далее возвращала ошибочные
4. Напиши интеграционный тест
5. Убедись, что даже при ошибках от сервиса, информация об игре возвращается
6. Настой параметры кэширования - TTL и объем используемой памяти
7. Вынеси эти настройки в файл с настройками Spring Boot
