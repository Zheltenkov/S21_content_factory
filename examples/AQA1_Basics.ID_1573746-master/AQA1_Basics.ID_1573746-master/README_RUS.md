## Глава 3
### 3.1. Что такое тест-раннер

Как свидетельствует само название, Test Runner — это инструмент, который помогает запускать или выполнять различные виды тестов, а также экспортировать собранные результаты в соответствии с заданными настройками. Это библиотека, которая запускает код теста, чтобы проверить наличие ошибок и багов. 

Если говорить простыми словами, тест-раннер как дирижёр в оркестре: организует процесс, запускает твои тесты и показывает, где что сломалось.
   
Тест-раннеры используются повсеместно в автоматизированном тестировании, будь то API, UI или мобильные приложения. Поэтому умение работать с ними — это базовый навык для любого инженера по автоматизированному тестированию.

С помощью тест-раннера ты можешь запускать тесты разными способами: передавая параметры запуска из командной строки или с помощью UI-инструментов.

Существует множество тест-раннеров, каждый из которых предназначен для работы с определённым языком программирования. Примеры самых популярных из них: JUnit для Java, Karma для JavaScript, PyTest для Python и т. д.

В этом и следующих проектах мы сосредоточимся на двух самых популярных тест-раннерах для Java: JUnit и TestNG. Они очень похожи по функциональности, но отличаются синтаксисом и некоторыми особенностями логики работы.

Функционал TestNG формировался довольно-таки линейно и изначально был шире, чем в версиях JUnit того же времени. Однако с выходом JUnit 5 этот тест-раннер совершил резкий скачок вперед и теперь предлагает достаточно широкий набор опций. JUnit 4 уже практически не используется в современных проектах, поэтому предлагаем тебе сразу изучать JUnit 5.

### 3.2. Исключения

В мире программирования ошибки и неожиданные ситуации, которые могут возникнуть во время работы программы, называются исключениями (exception). 

Эти исключения могут появляться по разным причинам: например, когда пользователь делает что-то не так, когда на диске отсутствует необходимый ресурс или когда теряется соединение с сервером.

Кроме того, исключения при выполнении программы могут возникать из-за ошибок в самом коде или неправильного использования API. В отличие от реальной жизни, программа должна точно знать, как действовать в такой ситуации. В Java для этого предусмотрен механизм обработки исключений.

Понимание, как работают исключения, нужны тебе не только для создания и настройки фреймворков автотестирования, но и могут сильно пригодиться во время собеседования — многие компании включают вопросы по этой теме в свои чек-листы для интервью на позицию инженера по автоматизированному тестированию. 

### 3.3. Что нужно знать о ключевых словах для обработки исключений в Java (try, catch, finally, throws)

Механизм обработки исключений в Java помогает справляться с ошибками и неожиданными ситуациями в коде. Давай разберёмся с основными ключевыми словами, которые используются для этого.

- try — определяет блок кода, в котором может произойти исключение;  
- catch — определяет блок кода, в котором происходит обработка исключения;  
- finally — определяет блок кода, который является необязательным, но при его наличии выполняется в любом случае независимо от результатов выполнения блока try (независимо от того, произошло исключение или нет).

Эти ключевые слова используются для создания в программном коде специальных обрабатывающих конструкций:

try{}catch \
try{}catch{}finally \
try{}finally{}

- throw — используется для возбуждения исключения;  
- throws — используется в сигнатуре методов и предупреждает о том, что метод может выбросить исключение.

QAA-инженеру умение использовать блоки try–catch и finally помогает создавать более надежные тесты и предотвращать возможные ошибки. 

### 3.4. Иерархия исключений Java

![hierarchy](./misc/images/hierarchy.png)
Источник: https://quarkphysics.ca/ICS4U1/unit2-FileIO/exceptions.html

Когда в программе возникает ошибка, исполняющая среда Java Virtual Machine (JVM) создает объект определенного типа из иерархии исключений Java. Эта иерархия представляет собой множество возможных исключительных ситуаций, унаследованных от общего «предка» — класса Throwable.

Неожиданные ситуации, возникающие в программе, можно разделить на две основные группы:

* ситуации, при которых восстановить дальнейшую нормальную работу программы невозможно;  
* ситуации, при которых восстановление работы программы возможно. 

К первой группе относятся ситуации, когда возникают исключения, унаследованные из класса Error. Эти ошибки возникают из-за серьёзных проблем, таких как сбои в работе JVM, переполнение памяти или сбой системы. Обычно они свидетельствуют о серьезных проблемах, устранить которые программными средствами невозможно. Такой вид исключений в Java считаются неконтролируемыми (unchecked) на стадии компиляции.

Также в эту группу входят RuntimeException — исключения, наследники класса Exception, генерируемые JVM во время выполнения программы. Часто причиной возникновения их являются ошибки программирования. Поскольку они тоже неконтролируемые (unchecked) на стадии компиляции, разработчики не обязаны писать код для их обработки.

Ко второй группе относятся исключительные ситуации, которые можно предвидеть еще на этапе написания программы. Для такого вида исключений должен быть написан код обработки.

Такие исключения считаются контролируемыми (checked). Основная задача разработчика при работе с исключениями в Java — это именно обработка таких ситуаций.

Понимание данной классификации помогает QA-инженеру предвидеть разные варианты поведения программы и заранее подготовить тесты для обработки различных ситуаций.
